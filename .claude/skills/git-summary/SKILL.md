- 変更内容を確認し、ワーキングツリーのステータスを確認。その中に、`.env`などの環境変数ファイルやAPIキー、その他の秘密情報が漏洩されていないかを確認してください。

- `git commit -m "<コミットメッセージ>"` を実行して、ステージングされた変更をコミット。

# 現在の状況確認

リポジトリの全体像を把握します。

- ワーキングツリーの状態: !`git status`
- ステージング済みの差分: !`git diff --cached`
- 未ステージングの差分: !`git diff`

# 指示

上記の変更内容を確認した上で、以下のステップに則って実行してください。

### 1. 秘匿情報のチェック（ゲート）

- ステージング済み・未ステージング・untrackedファイルの中身まで含めた全変更を対象に、`.env`系ファイル、APIキー、トークン、秘密鍵などの秘匿情報が含まれていないかを確認してください。
- 秘匿情報が疑われるファイルが見つかった場合は、そのファイルは絶対にステージングせず、該当ファイル・箇所を具体的にユーザーへ報告してそこで停止してください。

### 2. ファイルのグループ化とステージング(git add)

- 変更内容を論理的なグループ（機能・コンポーネント単位）に分類してください。
- 一見同じファイル群に見えても実装内容が異なる変更（無関係なルート修正など）は、diffの中身を実際に確認した上で別グループとして扱ってください。判断が曖昧な場合は根拠を示してください。
- グループごとに`git add <ファイル名>`でステージングしてください。

### 3. コミット(git commit)

- グループが1つのみで、かつユーザーが引数(`$ARGUMENTS`)としてコミットメッセージを指定している場合は、その文字列を最優先で使用してください。
- グループが複数になった場合、`$ARGUMENTS`は全体の文脈ヒントとしてのみ使い、各グループごとにConventional Commits（`feat:`, `fix:`, `chore:`等）に沿ったメッセージを個別に自動生成してください。複数コミットに分かれる場合はその旨を一言ユーザーに伝えてください。
- グループごとに`git commit -m "<コミットメッセージ>"`を実行してください。

### 4. リモートへの反映(git push) — 要確認

- 全コミット完了後、作成したコミット一覧（ハッシュ・メッセージ）を要約してユーザーに提示してください。
- pushは共有リモートへの反映であり取り消しにくい操作です。ユーザーの明示的な許可を得るまでpushを実行しないでください。
- 許可が得られたら、現在のブランチにリモート追跡ブランチが設定されているか確認してください（`git rev-parse --abbrev-ref --symbolic-full-name @{u}`）。未設定なら`git push -u origin <ブランチ名>`、設定済みなら`git push`を実行してください。
- push完了後、反映内容の要約を表示してください。
